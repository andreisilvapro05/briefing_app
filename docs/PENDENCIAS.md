# Pendências do sistema — auditoria automatizada

> Gerado por auditoria multi-agente (9 subsistemas) com verificação adversarial. 45 achados confirmados de 46 brutos.

## Severidade ALTA (12)

### A1. Credenciais default embutidas no código concedem admin/acesso-cliente em produção se a env faltar
- **Arquivo:** `src/lib/env.ts:70` · _auth_ · auth/segredo-default
- **Problema:** getServerEnv() faz `adminPassword: process.env.ADMIN_PASSWORD ?? "fysi-2026"` (linha 70) e `clientAccessCode: process.env.CLIENT_ACCESS_CODE ?? "fysi"` (linha 74). São segredos publicamente conhecidos (estão no repositório). Se qualquer deploy esquecer de setar ADMIN_PASSWORD/CLIENT_ACCESS_CODE, o app sobe com a senha default sem erro. isPasswordValid()/codeMatches() comparam contra esse default, e não há guarda de NODE_ENV=production que rejeite o fallback.
- **Cenário de falha:** Deploy no Vercel sem ADMIN_PASSWORD definido. Um atacante faz POST /api/auth/admin-password com {"password":"fysi-2026"} -> isPasswordValid retorna true -> setAdminSessionCookie() emite cookie fysi-admin válido por 30 dias -> acesso total ao painel admin (criar/editar clientes, contratos, uploads, webhooks). Idem: POST /api/auth/login com code "fysi" enumera PII de qualquer cliente pelo WhatsApp.
- **Fix sugerido:** Remover os defaults e exigir a env via required() dentro de getServerEnv (ou lançar erro quando NODE_ENV==='production' e a env estiver ausente). Nunca usar valor de fallback conhecido para segredo de autenticação.

### A2. Rota publica /api/cliente/contrato permite sequestro de cadastro de cliente via dedup por WhatsApp
- **Arquivo:** `src/app/api/cliente/contrato/route.ts:94` · _contracts_ · auth
- **Problema:** A rota POST e totalmente publica (nenhuma verificacao de sessao/ownership) e usa o service-role client. No branch sem clientId, o dedup identifica um cliente existente APENAS pelos digitos do WhatsApp (linha 97 compara `c.whatsapp` normalizado; o `nome` nao entra no match). Quando um cliente e casado por telefone, o codigo sobrescreve incondicionalmente TODOS os dados dele com o input nao autenticado (linhas 143-168: email, cpf, rg, cnpj, endereco, cep, razao_social). Pior: se o cliente casado ainda nao tinha email (coluna email e nullable desde a migration 20260502130000, pois clientes sao criados so com nome+whatsapp), `emailWasEmpty` fica true e a rota dispara `service.auth.signInWithOtp({ email: <email do atacante>, shouldCreateUser: true, data: { client_id: <id da vitima> } })` (linhas 217-234), enviando um magic link para o email do atacante ja vinculado ao client_id da vitima.
- **Cenário de falha:** Atacante que conhece o numero de WhatsApp de um cliente (semi-publico) faz POST { nome:'x', whatsapp:'<telefone da vitima>', email:'atacante@x.com', empresa:'x', endereco:'x', cep:'x', cpf:'x', como_conheceu:'x' }. O dedup casa a vitima pelo telefone, sobrescreve o email/CPF/endereco da vitima pelo do atacante e — se a vitima nunca setou email — envia um magic link ao atacante amarrado ao client_id da vitima, abrindo caminho para tomada de conta / acesso ao painel da vitima. Mesmo com email pre-existente, o atacante ainda vandaliza/rouba todo o PII da vitima.
- **Fix sugerido:** No branch de cliente EXISTENTE (casado por whatsapp ou via clientId), exigir prova de posse (sessao autenticada / capability token) antes de mutar o registro; nunca sobrescrever email/PII de um cliente ja existente com input anonimo; e nao disparar signInWithOtp com shouldCreateUser vinculando client_id a partir de uma atualizacao nao autenticada. Criar novo cliente so quando nao ha match.

### A3. Evento 'cobranca.paga' nunca chega ao app-financeiro: falta em KNOWN_EVENTS do receiver
- **Arquivo:** `src/app/api/fysi/webhook/route.ts:20` · _cobrancas_ · data-flow-inconsistency
- **Problema:** O tipo DashboardEvent (dashboard-webhook.ts:18-23) e o payload CobrancaPagaPayload incluem 'cobranca.paga', e actions.ts dispara esse evento em registrarPagamentoAction e marcarPagoEsteMesAction via sendCobrancaPagaWebhook. Porém o array KNOWN_EVENTS do receiver (route.ts:20-25) lista apenas cliente.criado/atualizado, contrato.assinado e pagamento.atualizado — 'cobranca.paga' está de fora. O próprio doc-comment do route.ts (linhas 12-14) afirma que DASHBOARD_WEBHOOK_URL aponta para o próprio briefing_app, que valida e ENCAMINHA pro app-financeiro. Como 'cobranca.paga' não está em KNOWN_EVENTS, o receiver responde 422 unknown-event (linha 88-94) e nunca faz o forward. Ou seja, o feature 'sync com app financeiro via webhook' (commit 875e67b) está quebrado exatamente para cobranças — o núcleo do subsistema.
- **Cenário de falha:** Admin clica 'Marcar pago hoje' numa cobrança mensal. actions.ts monta o payload event='cobranca.paga' e faz POST para DASHBOARD_WEBHOOK_URL (= o próprio /api/fysi/webhook). O receiver valida o HMAC com sucesso, mas KNOWN_EVENTS.includes('cobranca.paga') === false, então retorna 422 {reason:'unknown-event'} e NÃO reenvia pro app-financeiro-lovat.vercel.app. O pagamento fica registrado no briefing_app mas o dashboard financeiro nunca recebe o evento — sincronização silenciosamente perdida.
- **Fix sugerido:** Adicionar "cobranca.paga" ao array KNOWN_EVENTS em route.ts (e conferir se o app-financeiro trata esse evento). Idealmente derivar KNOWN_EVENTS do próprio tipo DashboardEvent para não divergirem de novo.

### A4. Segredo HMAC do app-financeiro hardcoded no código versionado
- **Arquivo:** `src/app/api/fysi/webhook/route.ts:33` · _cobrancas_ · security-secret-exposed
- **Problema:** RECEIVER_SECRET (o FYSI_WEBHOOK_SECRET que o app-financeiro usa pra validar assinaturas) está escrito em texto claro no código-fonte e commitado no git (linhas 33-34), com comentário admitindo a prática. Qualquer pessoa com acesso de leitura ao repositório — colaboradores, forks, vazamento — obtém o segredo compartilhado usado pra assinar webhooks aceitos pelo app-financeiro. Com ele é possível forjar eventos válidos (pagamento.atualizado, contrato.assinado etc.) diretamente contra https://app-financeiro-lovat.vercel.app/api/webhooks/fysi, poluindo/fraudando os dados financeiros.
- **Cenário de falha:** Um dev com acesso ao repo (ou alguém que clone um fork/histórico) copia a constante de 64 hex chars, calcula HMAC-SHA256 de um corpo arbitrário e faz POST assinado direto no receiver do app-financeiro, injetando pagamentos falsos sem passar pelo briefing_app nem por qualquer auth de admin.
- **Fix sugerido:** Mover o segredo para uma env var (ex: FYSI_RECEIVER_SECRET) lida via getServerEnv, remover a constante do código e ROTACIONAR o segredo nos dois projetos, já que ele deve ser considerado comprometido por ter estado no histórico do git.

### A5. localStorage global não-namespaced + merge 'local vence' vaza e corrompe briefing entre clientes
- **Arquivo:** `src/lib/briefing-store.ts:68` · _briefing_ · inconsistencia-fluxo-de-dados
- **Problema:** O STORAGE_KEY é único e global ('fysi.briefing.responses.v1', linha 12) — não é namespaced por clientId. pullResponsesFromServer faz writeAll({ ...data.responses, ...local }) (linha 68), então as respostas locais SEMPRE vencem as do servidor. O fluxo /entrar (src/app/entrar/page.tsx:65-67) chama hydrateCliente(data) e pullResponsesFromServer(data.id) SEM chamar clearAllResponses() antes. clearAllResponses só roda no 'Começar novo' (app/page.tsx:72) e no 'Sair' (dashboard/page.tsx:186), que o cliente pode nunca acionar.
- **Cenário de falha:** Computador compartilhado da agência (cenário 'sócio convidado' que o próprio código diz suportar): Cliente A preenche o briefing (localStorage cheio das respostas de A) e fecha a aba sem clicar 'Sair'. Cliente B abre /entrar (link do e-mail), loga com o WhatsApp+código dele. pullResponsesFromServer(B) traz poucas/nenhuma resposta do servidor; writeAll({...serverB, ...localA}) faz os valores de A sobrescreverem. O dashboard/revisão de B agora exibe as respostas privadas de A (vazamento). Ao B editar qualquer campo, o autosave persiste (save route) sob o clientId de B — gravando os dados de A no registro de B no banco (corrupção).
- **Fix sugerido:** Namespacear o STORAGE_KEY por clientId (ex.: `fysi.briefing.responses.v1.<clientId>`) OU chamar clearAllResponses() em /entrar antes de hydrateCliente/pull. Além disso, inverter a precedência do merge para respostas do servidor de um clientId recém-carregado (server vence) quando o clientId em localStorage mudou.

### A6. IDOR: /api/me/stage devolve contrato_signed_url + entrega + PII a quem só conhece o clientId (obtido via nome+whatsapp)
- **Arquivo:** `src/app/api/me/stage/route.ts:64` · _client_ · auth/permissao
- **Problema:** O endpoint é totalmente sem autenticação: aceita { clientId } no body e responde com contrato_signed_url (link do PDF do contrato ASSINADO no Autentique, que contém nome completo, CPF, RG, endereço), entrega_documento, fysi_drive_link, pagamento_total/pago e observações. O comentário do arquivo (linhas 10-14) afirma explicitamente 'a resposta não inclui PII' — o que é FALSO. Pior: o clientId não é um segredo. /api/auth/start (app/api/auth/start/route.ts) faz dedup por whatsapp normalizado e RETORNA o clientId de um cliente já existente (linha 196/199) para qualquer chamador que informe nome+whatsapp da vítima, sem nenhuma verificação de posse do número. Ou seja, o clientId é tratado como bearer secret mas é distribuído por um endpoint público keyed em telefone.
- **Cenário de falha:** Atacante sabe o nome e o WhatsApp de um cliente da Fysi. 1) POST /api/auth/start { nome, whatsapp } → recebe clientId da vítima. 2) POST /api/me/stage { clientId } → recebe contrato_signed_url, abre o PDF do contrato assinado com CPF/RG/endereço da vítima, além de dados de pagamento e link do Drive. Vazamento de PII sensível sem nenhuma credencial.
- **Fix sugerido:** Exigir sessão Supabase autenticada (magic link) nos endpoints /api/me/*: derivar o clientId do JWT/claim (user_metadata.client_id) em vez de aceitá-lo do body, e conferir que bate com o registro. Nunca retornar contrato_signed_url/entrega_documento sem auth. Se manter o modelo por clientId por ora, no mínimo parar de expor contrato_signed_url e entrega_documento aqui e remover a afirmação falsa 'não inclui PII'. E /api/auth/start não deve devolver o clientId de um cliente já existente sem verificação do telefone.

### A7. Sequestro de conta / sobrescrita de PII: /api/cliente/contrato público faz dedup por whatsapp sem auth e dispara magic-link para o e-mail do atacante
- **Arquivo:** `src/app/api/cliente/contrato/route.ts:94` · _client_ · auth/permissao
- **Problema:** No modo público (sem clientId), o handler carrega todos os clients e casa por whatsapp normalizado (linhas 94-99). Se encontra um registro existente (found), passa a atualizar esse registro com o email/empresa/endereco/cpf/cnpj vindos do payload (linhas 143-168) — sem qualquer verificação de que o chamador é dono do número. Além da sobrescrita destrutiva dos dados de contrato da vítima, se o registro da vítima ainda não tinha e-mail (caso comum: fez só a Tela 1 com nome+whatsapp), emailWasEmpty=true e o código dispara signInWithOtp para parsed.email (e-mail do ATACANTE) carregando client_id = id da vítima nos metadados (linhas 217-234). O atacante recebe um magic-link que, ao ser confirmado em /auth/callback, o associa ao painel da vítima.
- **Cenário de falha:** Cliente Maria fez a Tela 1 (nome + whatsapp, sem e-mail). Atacante abre /contratar, preenche nome/whatsapp de Maria + o próprio e-mail e um CPF qualquer, confirma. O endpoint acha o registro de Maria pelo whatsapp, sobrescreve os dados de contrato e envia magic-link ao e-mail do atacante com client_id de Maria → atacante entra no painel de Maria (contrato, pagamentos, entregas).
- **Fix sugerido:** Não permitir que o fluxo público sobrescreva um registro pré-existente casado só por whatsapp. Para clientes já existentes exigir clientId + sessão autenticada. O magic-link nunca deve ser emitido para um e-mail arbitrário vinculado ao client_id de um registro que o chamador não provou possuir; verificar posse do whatsapp (OTP SMS) antes de associar, ou só emitir OTP quando o registro foi realmente criado agora (isNewClient).

### A8. Segredo HMAC hardcoded no código-fonte (RECEIVER_SECRET)
- **Arquivo:** `src/app/api/fysi/webhook/route.ts:34` · _admin_ · seguranca
- **Problema:** O segredo usado para assinar (HMAC-SHA256) os webhooks reenviados ao app-financeiro está hardcoded em texto claro no repositório: `const RECEIVER_SECRET = "eaf930b8...cda97"`. É o mesmo valor de FYSI_WEBHOOK_SECRET no receiver do app-financeiro. Esse endpoint é alimentado pelas ações admin (setPaymentAction, mark-signed, refresh, cobrancas) via sendDashboardWebhook.
- **Cenário de falha:** Qualquer pessoa com acesso de leitura ao repositório (colaborador, fork, vazamento de código, histórico git) obtém o segredo e pode forjar requisições assinadas diretamente para https://app-financeiro-lovat.vercel.app/api/webhooks/fysi — injetando eventos falsos de pagamento.atualizado / contrato.assinado / cobranca.paga no dashboard financeiro, corrompendo os números de receita. Rotacionar o segredo exige deploy em duas contas Vercel separadas.
- **Fix sugerido:** Mover o segredo para uma env var (ex.: FYSI_RECEIVER_SECRET) lida via getServerEnv(), remover o literal do código e rotacionar o segredo atual (já deve ser considerado comprometido por estar no histórico git).

### A9. Segredo do webhook do app-financeiro hardcoded no código-fonte
- **Arquivo:** `src/app/api/fysi/webhook/route.ts:33` · _api_ · security
- **Problema:** RECEIVER_SECRET (o HMAC secret que autentica o briefing_app perante o receiver do app-financeiro) está escrito em texto puro no arquivo e commitado no git (`const RECEIVER_SECRET = "eaf930b83fdf..."`). Qualquer pessoa com acesso ao repositório (colaborador, fork, vazamento) obtém a chave que assina eventos financeiros.
- **Cenário de falha:** Um atacante que leu o repo faz POST direto para https://app-financeiro-lovat.vercel.app/api/webhooks/fysi com body `{"event":"pagamento.atualizado",...}` e o header X-Fysi-Signature = HMAC-SHA256(body, RECEIVER_SECRET). O receiver valida a assinatura como legítima e injeta pagamentos/contratos falsos no dashboard financeiro. Rotacionar a chave é impossível sem novo deploy dos dois apps.
- **Fix sugerido:** Mover para uma env (ex. FYSI_RECEIVER_SECRET) lida via getServerEnv(); remover o valor do histórico do git e rotacionar o segredo nos dois projetos Vercel.

### A10. Defaults fracos e commitados para ADMIN_PASSWORD e CLIENT_ACCESS_CODE, sem guard em produção
- **Arquivo:** `src/lib/env.ts:70` · _api_ · auth
- **Problema:** getServerEnv() define `adminPassword: process.env.ADMIN_PASSWORD ?? "fysi-2026"` (linha 70) e `clientAccessCode: process.env.CLIENT_ACCESS_CODE ?? "fysi"` (linha 74). Diferente de serviceRoleKey/supabaseUrl, esses NÃO passam por required() — se as envs não forem setadas, getServerEnv() não falha; o app sobe silenciosamente com os defaults. Os comentários dizem 'SEMPRE setar em produção' mas nada impede o contrário.
- **Cenário de falha:** Deploy em produção sem ADMIN_PASSWORD definido: o painel /admin fica destravado com a senha pública 'fysi-2026' (login por senha em /api/auth/admin-password e via ?key=fysi-2026 em todas as rotas admin). Sem CLIENT_ACCESS_CODE, /api/auth/login aceita o código 'fysi' (4 chars) para qualquer telefone, devolvendo o clientId — que é a credencial bearer de todo o resto do app.
- **Fix sugerido:** Em produção (isProduction()) exigir ADMIN_PASSWORD e CLIENT_ACCESS_CODE via required() ou lançar erro no boot; nunca usar default em NODE_ENV=production.

### A11. RLS nunca habilitada em admin_notifications — tabela exposta via anon key
- **Arquivo:** `supabase/migrations/20260602000000_add_admin_notifications.sql:8` · _db_ · auth-rls
- **Problema:** A migration cria public.admin_notifications mas nunca executa `alter table ... enable row level security` nem cria policy alguma (grep em supabase/ confirma que só clients/briefing_responses/briefing_files têm `enable row level security`). No Supabase, tabelas do schema public recebem grants default para os roles anon/authenticated; com RLS desabilitada o PostgREST libera SELECT/INSERT/UPDATE/DELETE para qualquer um que tenha a anon key — que é pública por design (NEXT_PUBLIC_SUPABASE_ANON_KEY, embutida no bundle do browser em src/lib/supabase/browser.ts). O app só acessa essa tabela via service role (src/app/admin/page.tsx, src/lib/notifications.ts), então a ausência de RLS não é notada em uso normal, mas o dado fica aberto na API REST.
- **Cenário de falha:** Um visitante anônimo pega a NEXT_PUBLIC_SUPABASE_URL e a anon key do JS público e faz GET {url}/rest/v1/admin_notifications?select=* — recebe todas as notificações (title/message com nome de cliente, client_id). Também pode DELETE/INSERT para apagar a fila ou injetar avisos falsos no /admin.
- **Fix sugerido:** Na migration adicionar `alter table public.admin_notifications enable row level security;` sem criar policies (nega tudo para anon/authenticated; o backend usa service_role que faz bypass). Aplicar também em produção via nova migration idempotente.

### A12. RLS nunca habilitada em cobrancas_mensais — dados financeiros e PII expostos via anon key
- **Arquivo:** `supabase/migrations/20260607000000_add_cobrancas_mensais.sql:9` · _db_ · auth-rls
- **Problema:** public.cobrancas_mensais é criada sem `enable row level security` e sem policies. A tabela guarda PII e dados financeiros: nome, empresa, whatsapp, email, valor_mensal, historico (jsonb com todos os pagamentos, formas, observações). Como toda tabela do public sem RLS no Supabase, fica totalmente acessível pela anon key pública. O app só a usa via service role (src/app/admin/cobrancas/actions.ts, relatorios/page.tsx), mascarando o buraco.
- **Cenário de falha:** Atacante anônimo faz GET {url}/rest/v1/cobrancas_mensais?select=* com a anon key e vaza toda a carteira de cobranças (receita mensal/MRR, contatos dos clientes, histórico de pagamentos). Pode ainda DELETE ou UPDATE valor_mensal, destruindo/adulterando o controle financeiro sem passar por nenhum admin.
- **Fix sugerido:** Adicionar `alter table public.cobrancas_mensais enable row level security;` em nova migration (sem policies, para negar anon/authenticated e permitir só service_role). Rodar `select ... get_advisors` para confirmar que o lint rls_disabled_in_public zerou.

## Severidade MEDIA (22)

### M1. E-mail do cliente não é normalizado, quebrando o vínculo auth_user_id no callback do magic link
- **Arquivo:** `src/app/auth/callback/route.ts:47` · _auth_ · fluxo-de-dados/correcao
- **Problema:** O callback liga a sessão Supabase ao registro do cliente com `.update({auth_user_id, email_verified_at}).eq("email", user.email).is("auth_user_id", null)` (linhas 41-48). Mas `user.email` vem do Supabase Auth, que armazena o e-mail sempre em minúsculas, enquanto os registros de `clients` são gravados com o e-mail VERBATIM (sem lowercase): api/auth/start linha 155 (`email: parsed.email ?? ""`) e api/cliente/contrato linhas 110 e 146 (`email: parsed.email`). O zod `.email()` não normaliza caixa. Só admin-login lowercaseia. Logo, e-mail com maiúsculas nunca casa no `.eq`.
- **Cenário de falha:** Cliente digita "Joao@Empresa.com" em /contrato -> gravado como "Joao@Empresa.com" em clients -> magic link cria auth.user com email "joao@empresa.com" -> callback executa update .eq("email","joao@empresa.com") que casa 0 linhas -> auth_user_id e email_verified_at ficam null permanentemente; o cliente autentica mas nunca é vinculado ao seu briefing. O mesmo desalinhamento de caixa também faz o dedup por e-mail (start:109, contrato) falhar e criar registros duplicados.
- **Fix sugerido:** Normalizar o e-mail para minúsculas na escrita (`parsed.email.toLowerCase().trim()`) em start/contrato e nas queries de dedup, e comparar sempre em lowercase no callback.

### M2. Senha do admin trafega como ?key= na URL (login por password), vazando via Referer/logs/histórico
- **Arquivo:** `src/lib/admin.ts:33` · _auth_ · seguranca/segredo-exposto
- **Problema:** getAdminUser aceita a própria ADMIN_PASSWORD como query param: `if (env.adminPassword && opts.urlKey === env.adminPassword)` (linha 33). A página de login redireciona explicitamente para `/admin?key=${encodeURIComponent(password)}` (src/app/admin/login/page.tsx:38) e todas as páginas admin propagam `?key=` em links e inputs hidden. O valor NÃO é um token derivado — é a senha em claro. Além da nota de 'não compartilhe', não há mitigação para vazamento automático via header Referer (qualquer recurso/link externo numa página admin envia a URL completa a terceiros), logs de servidor/proxy e histórico do navegador. A comparação também é `===` (não time-constant), diferente do caminho por cookie.
- **Cenário de falha:** Admin abre /admin?key=SENHA. A página contém um link/imagem para domínio externo (ou o admin clica num link externo) -> o navegador envia Referer: https://app/admin?key=SENHA para o terceiro -> a senha admin real é capturada nos logs desse terceiro, concedendo acesso permanente ao painel.
- **Fix sugerido:** Não usar a senha crua na URL. Se o fallback sem-cookie é necessário, emitir um token opaco derivado (como o do cookie fysi-admin) e comparar em time-constant; setar Referrer-Policy: no-referrer nas páginas admin; e comparar urlKey com timingSafeEqual.

### M3. Login do cliente e dedup carregam a tabela clients inteira em memória (scan sem filtro/limite)
- **Arquivo:** `src/app/api/auth/login/route.ts:76` · _auth_ · eficiencia/escalabilidade
- **Problema:** O login do cliente faz `.from("clients").select("id,nome,email,empresa,whatsapp,project_type").order("created_at")` sem WHERE nem limit (linhas 76-79) e filtra o WhatsApp em JS com Array.find (linhas 84-87). O mesmo padrão de scan total ocorre no dedup por WhatsApp em api/auth/start (linhas 117-124) e em api/cliente/contrato (linhas 94-99). Como o WhatsApp é armazenado com formatação variável, optaram por normalizar em JS — mas isso puxa todos os registros (com PII) para o servidor a cada login/registro.
- **Cenário de falha:** Com a base crescendo para milhares de clientes, cada POST /api/auth/login (endpoint público, protegido só pelo código global compartilhado) materializa a tabela inteira no Node: latência e uso de memória crescem O(n) e o endpoint vira vetor de exaustão de recursos sob tentativas repetidas.
- **Fix sugerido:** Persistir uma coluna normalizada `whatsapp_digits` (só dígitos) com índice e consultar `.eq("whatsapp_digits", digits).limit(1)`, eliminando o scan total e o filtro em JS nos três pontos.

### M4. mark-signed dispara webhook contrato.assinado incondicionalmente (evento duplicado)
- **Arquivo:** `src/app/api/admin/contracts/mark-signed/[id]/route.ts:70` · _contracts_ · data-flow
- **Problema:** Diferente da rota refresh — que guarda o disparo com `newStatus === 'assinado' && previousStatus !== 'assinado'` (refresh/[id]/route.ts linha 76) — a rota mark-signed NAO le o status anterior e SEMPRE envia o webhook `contrato.assinado` sempre que `fresh` existe (linhas 70-90). ### B1. Callback expõe mensagem de erro crua do Supabase na URL de redirect
- **Arquivo:** `src/app/auth/callback/route.ts:32` · _auth_ · seguranca/vazamento-de-erro
- **Problema:** Em falha do exchangeCodeForSession o handler redireciona para `/auth/erro?reason=${encodeURIComponent(error.message)}` (linhas 30-34), refletindo a mensagem interna do provedor de auth diretamente na query string exposta ao usuário e a logs.
- **Cenário de falha:** Um code inválido/expirado produz um redirect como /auth/erro?reason=invalid%20request%3A%20... com detalhes internos do Supabase Auth, que ficam em histórico do navegador e logs de acesso e podem auxiliar reconhecimento de detalhes do backend.
- **Fix sugerido:** Redirecionar com um código de motivo genérico e estável (ex.: ?reason=auth) e logar a mensagem detalhada apenas server-side via logServerError.

### B2. Status 'cancelado' presente na UI mas nunca setado por nenhuma rota (dead / feature incompleta)
- **Arquivo:** `src/components/admin/contract-card.tsx:38` · _contracts_ · dead-code
- **Problema:** STATUS_LABELS e STATUS_TONES mapeiam 'cancelado' (linhas 38, 44), mas nenhuma rota do subsistema jamais grava contrato_status='cancelado' — send grava 'pendente', mark-signed grava 'assinado', refresh grava 'rejeitado'|'assinado'|'pendente'. Alem disso o status 'rejeitado' computado no refresh (refresh/[id]/route.ts linha 56) e persistido mas nunca dispara webhook/notificacao (so 'assinado' dispara), diferente do fluxo de assinatura.
- **Cenário de falha:** Um contrato cancelado no Autentique nunca reflete 'cancelado' no painel (o refresh so mapeia rejeitado/assinado/pendente), e uma rejeicao do cliente atualiza o status mas nao notifica ninguem nem avisa o dashboard financeiro — indicando fluxo de estado meio-implementado.
- **Fix sugerido:** Ou remover o label 'cancelado' morto, ou implementar deteccao de cancelamento no refresh; e considerar disparar notificacao/webhook quando o status virar 'rejeitado'.

### B3. Edição de cobrança não persiste whatsapp/email/tipo/data_vencimento (edição meio-implementada)
- **Arquivo:** `src/app/admin/cobrancas/actions.ts:69` · _cobrancas_ · half-implemented-feature
- **Problema:** updateCobrancaAction só lê e grava nome, empresa, descricao, valor_mensal, dia_cobranca e ativa. Os campos whatsapp, email, tipo e data_vencimento — todos setáveis na criação (addCobrancaAction) — não têm nenhum input no formulário de edição (page.tsx:484-539) nem são tratados na action, então são impossíveis de corrigir depois. Como whatsapp/email compõem o payload cobranca.paga (actions.ts:200-212), um contato digitado errado na criação é propagado permanentemente pro app-financeiro sem forma de conserto pela UI; e uma pontual com data de vencimento errada não pode ser ajustada.
- **Cenário de falha:** Admin cria cobrança com email errado (typo). Depois abre 'Editar / inativar', mas não há campo de email; salvar mantém o email errado. Todo webhook cobranca.paga subsequente envia o email incorreto ao dashboard financeiro, e a única alternativa é apagar e recriar a cobrança (perdendo o histórico).
- **Fix sugerido:** Adicionar os campos faltantes (whatsapp, email e, para pontuais, tipo/data_vencimento) ao formulário de edição e ao Record de updates em updateCobrancaAction.

### B4. statusDoMes marca 'atrasado' já no próprio dia de cobrança (off-by-one)
- **Arquivo:** `src/lib/cobrancas-mensais.ts:102` · _cobrancas_ · off-by-one
- **Problema:** Para cobrança mensal não paga, statusDoMes faz 'return hoje < c.dia_cobranca ? a_cobrar : atrasado'. No dia exatamente igual a dia_cobranca (hoje === dia_cobranca), a condição hoje < dia é falsa, então a cobrança já é classificada como 'atrasado' no próprio dia do vencimento, antes de o admin ter o dia inteiro pra cobrar. O esperado é que o dia da cobrança ainda conte como 'a_cobrar'.
- **Cenário de falha:** Cobrança com dia_cobranca=10, sem pagamento no mês. No dia 10 o painel já mostra a cobrança em vermelho no KPI 'Em atraso' e no filtro 'Atrasadas', enquanto o correto seria 'a cobrar' até o fim do dia 10.
- **Fix sugerido:** Usar 'hoje <= c.dia_cobranca ? "a_cobrar" : "atrasado"' para que o dia do vencimento ainda conte como a cobrar.

### B5. Indicador 'Salvando…/Salvo' do cabeçalho do bloco fica permanentemente em 'idle'
- **Arquivo:** `src/app/briefing/[bloco]/page.tsx:92` · _briefing_ · feature-incompleta
- **Problema:** BlocoShell recebe saveStatus com default 'idle' (bloco-shell.tsx:32) e renderiza o SaveStatusPill a partir dele, mas a página [bloco]/page.tsx nunca passa a prop saveStatus, e o status por-campo do useBriefingField é interno a cada componente e nunca é elevado ao shell. Resultado: a pílula de status no cabeçalho mostra sempre '—' (idle), nunca 'Salvando…' nem 'Salvo'.
- **Cenário de falha:** Cliente preenche campos; o autosave ocorre, mas o único indicador visível de salvamento na tela (a pílula do cabeçalho) permanece '—' o tempo todo, não dando feedback de que o trabalho foi salvo — feature meio-implementada.
- **Fix sugerido:** Elevar o SaveStatus agregado dos campos do bloco (ex.: via contexto ou callback) e passá-lo como saveStatus para BlocoShell, ou remover a pílula se o feedback for por-campo.

### B6. Listener de postMessage do Calendly sem checagem de origin permite marcar chamada como agendada
- **Arquivo:** `src/app/agendar/page.tsx:53` · _client_ · validacao/seguranca
- **Problema:** handleMessage aceita qualquer MessageEvent cujo data.event === 'calendly.event_scheduled' e chama markAsScheduled(), sem verificar e.origin (deveria ser https://calendly.com). Qualquer janela/iframe/extensão que faça window.postMessage com esse shape marca a chamada como agendada (POST /api/cliente/chamada action:'agendou').
- **Cenário de falha:** Uma página ou script de terceiros com referência à janela envia postMessage({event:'calendly.event_scheduled'}); a chamada do cliente é marcada como agendada sem que nenhum agendamento real tenha ocorrido, quebrando o estado do painel e a lógica de próximos passos.
- **Fix sugerido:** Validar e.origin === 'https://calendly.com' (e opcionalmente a presença de payload.event.uri) antes de agir no evento.

### B7. Allowlist de MIME é ignorada quando o Content-Type do arquivo vem vazio
- **Arquivo:** `src/app/api/upload/route.ts:82` · _api_ · validation
- **Problema:** A verificação é `if (file.type && !isMimeAllowed(file.type))` (upload linha 82; mesmo padrão em transcribe linha 55 com `audio.type &&`). Quando file.type é string vazia, o curto-circuito pula a checagem inteira e o upload é aceito independentemente do tipo real.
- **Cenário de falha:** Cliente envia um arquivo com Content-Type removido/vazio (trivial de forjar no FormData). Ele passa o gate de MIME e é gravado no bucket público (getPublicUrl), permitindo hospedar conteúdo arbitrário sob o domínio do storage do projeto, dentro do limite de 25 MB.
- **Fix sugerido:** Tratar tipo ausente/vazio como não permitido (ou inferir o MIME a partir da extensão/magic bytes) antes de aceitar; aplicar o mesmo em transcribe.

### B8. CHECK de contrato_status ausente no setup-completo (divergência de enum entre as duas fontes de schema)
- **Arquivo:** `supabase/setup-completo.sql:86` · _db_ · schema-drift
- **Problema:** A migration 20260525000000_add_contract_columns.sql define contrato_status com CHECK (null ou in ('pendente','assinado','rejeitado','cancelado')). Já o setup-completo.sql cria a mesma coluna como `add column if not exists contrato_status text` sem nenhum CHECK. As duas fontes de verdade do schema divergem: um banco criado pela migration valida o enum, um criado pelo setup-completo aceita qualquer string.
- **Cenário de falha:** Banco provisionado via setup-completo.sql não rejeita valores inválidos: um bug/typo gravando contrato_status='enviada' ou 'assinadoo' passa silenciosamente, e a UI do /admin (que faz switch sobre os valores canônicos) cai no ramo default/estado desconhecido em vez de falhar cedo. O mesmo dado seria rejeitado no banco criado pela migration, mascarando o bug em um ambiente e não no outro.
- **Fix sugerido:** Replicar o CHECK de contrato_status no setup-completo.sql (ou eliminar o setup-completo e gerar sempre a partir das migrations) para manter as duas definições idênticas.

### B9. Upload não-autenticado permitido apenas com clientId conhecido, sem rate-limit
- **Arquivo:** `src/app/api/upload/route.ts:117` · _integrations_ · auth
- **Problema:** No caminho (2) (linhas 117-131), sem sessão Supabase, basta enviar um clientId (UUID) válido no FormData para gravar arquivos no Storage e inserir linhas em briefing_files usando o service-role client (que ignora RLS). Não há rate-limit nem prova de posse do cliente — só a existência do UUID na tabela clients.
- **Cenário de falha:** Alguém que obtenha/vaze o UUID de um cliente (aparece em links, e-mails, etc.) pode enviar repetidamente arquivos de até 25MB atribuídos àquele cliente, poluindo o Storage, criando registros em briefing_files e atualizando last_client_activity_at (mascarando o indicador de cliente 'parado' no admin).
- **Fix sugerido:** Exigir ao menos um token/segredo ligado ao magic_slug do cliente para o caminho pré-auth, ou aplicar rate-limit por IP/clientId, ou restringir esse caminho apenas à etapa de briefing ainda não autenticada com TTL.

### B10. Allowlist de MIME é ignorada quando Content-Type vem vazio
- **Arquivo:** `src/app/api/upload/route.ts:82` · _integrations_ · validacao
- **Problema:** A checagem é `if (file.type && !isMimeAllowed(file.type))`. Quando file.type é string vazia (upload sem Content-Type), a condição faz curto-circuito e o arquivo passa sem validação de tipo. Mesmo padrão em transcribe/route.ts linha 55 (`audio.type && !audio.type.startsWith('audio/')`).
- **Cenário de falha:** Cliente/atacante envia multipart com Content-Type vazio para /api/upload; o arquivo é aceito independentemente do tipo real (fica gravado como application/octet-stream). Em /api/transcribe, um arquivo não-áudio com type vazio é encaminhado ao Whisper, gerando custo em USD para conteúdo inválido.
- **Fix sugerido:** Tratar MIME ausente como rejeição (ou exigir tipo explícito): `if (!file.type || !isMimeAllowed(file.type)) return errorResponse('unsupported-mime', 415)`.

### B11. Vocabulário de categorias divergente: subfolderForField não cobre áudios nem documentos
- **Arquivo:** `src/lib/google-drive.ts:43` · _integrations_ · inconsistencia-fluxo
- **Problema:** file-categories.ts define 7 categorias incluindo 'audios' e 'documentos', mas subfolderForField (usado pelo /api/upload para escolher a subpasta do Drive, linha 202) só mapeia logo/identidade/imagens/depoimentos e joga todo o resto em '06-Materiais do cliente'. Áudios e documentos (PDF, docx) do cliente caem numa pasta genérica no Drive enquanto na UI aparecem categorizados corretamente.
- **Cenário de falha:** Cliente anexa uma gravação de áudio e um PDF. Na UI do admin (categorizeFile) aparecem em 'Áudios' e 'Documentos'; no Google Drive espelhado ambos vão parar em '06-Materiais do cliente', divergindo da organização mostrada ao time e dificultando achar o arquivo.
- **Fix sugerido:** Alinhar subfolderForField com as categorias de file-categories.ts (adicionar subpastas/branches para áudio e documentos via mime/field_id) ou derivar a subpasta a partir de categorizeFile para ter uma única fonte de verdade.
